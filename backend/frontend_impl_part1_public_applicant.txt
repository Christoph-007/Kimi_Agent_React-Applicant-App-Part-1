### SCR-01 — Landing Page (`/`)

#### A. Design Elements
| Property | Value |
|----------|-------|
| Layout | Single-column sections (full-width hero → stats → how-it-works → featured jobs grid → testimonials → footer) |
| Primary Colors | Brand indigo `#4F46E5`, Accent amber `#F59E0B`, bg white / dark-gray |
| Typography | H1: `text-5xl font-bold`, H2: `text-3xl font-semibold`, body: `text-base text-gray-600` |
| Spacing | Section padding `py-20 px-6`, max width `max-w-7xl mx-auto` |
| Responsive | Desktop: 3-col job grid; Tablet: 2-col; Mobile: 1-col stack, hero text scales down |
| Dark Mode | Class-based `dark:` variants on bg and text |

#### B. UI Components
| Component | Type | States |
|-----------|------|--------|
| `Navbar` | Reusable | Default, scrolled (shadow), mobile (hamburger menu) |
| `HeroSection` | Screen-specific | Static |
| `StatsBar` | Screen-specific | Loading skeleton, loaded |
| `HowItWorksCard` | Screen-specific | Static |
| `JobCard` (×6) | Reusable | Default, hover (shadow lift), loading skeleton |
| `TestimonialCard` | Screen-specific | Static |
| `Footer` | Reusable | Static |
| `Button` | Reusable | Primary, secondary, hover, disabled |

#### C. Functional Elements
- **Interactions:** Hero CTA buttons navigate to `/jobs` and `/signup/employer`; Job cards navigate to `/jobs/:id`
- **API Calls:** `GET /api/jobs?limit=6&status=open` — loads featured jobs on mount
- **Loading State:** 6 skeleton `JobCard` placeholders while fetching
- **Error State:** Toast notification; fallback to hardcoded placeholder cards
- **Empty State:** N/A — always show 6 jobs or skeletons
- **Permissions:** No auth required; Navbar shows "Login" + "Sign Up" if unauthenticated

#### D. Navigation & Routing
- **Route:** `/`
- **Guard:** None
- **Redirect:** If authenticated, Navbar links update to portal-specific items (no forced redirect)

#### E. Implementation Notes
- **Folder:** `src/pages/public/LandingPage.jsx`
- **Libraries:** None specific; use CSS animation for hero entrance
- **Performance:** Lazy-load testimonials and footer sections
- **Accessibility:** `<main>`, `<section aria-label="...">`, all CTAs have descriptive text

### SCR-02 — Login (`/login`) ⚠️ Auth Sensitive

#### A. Design Elements
| Property | Value |
|----------|-------|
| Layout | Centered card, split (left: branding illustration, right: form) on desktop; single column mobile |
| Colors | White card, indigo submit button, red error text |
| Typography | Card title `text-2xl font-bold`, labels `text-sm font-medium` |
| Spacing | Form fields `gap-4`, card `p-8 rounded-2xl shadow-lg` |
| Responsive | Split → stacked below `md` breakpoint; illustration hidden on mobile |

#### B. UI Components
| Component | Type | States |
|-----------|------|--------|
| `AuthLayout` | Reusable wrapper | Static |
| Radio group (user type) | Screen-specific | Default, selected ring |
| `TextInput` (email/phone) | Reusable | Default, focused, error, disabled |
| `PasswordInput` (with toggle) | Reusable | Default, focused, visible/hidden |
| `Button` (Login) | Reusable | Default, loading spinner, disabled |
| `FormError` | Reusable | Shown on API error |

#### C. Functional Elements
- **Validation:** All fields required; password min 6 chars; email/phone format check
- **Submission:** `POST /api/auth/login { identifier, password, userType }` → store JWT in `localStorage` / `AuthContext`
- **Post-login redirect:** `employer` → `/employer/dashboard`; `applicant` → `/applicant/dashboard`; `admin` → `/admin/dashboard`
- **Loading:** Button shows spinner, inputs disabled during request
- **Error:** API error message shown below form (e.g., "Invalid credentials")
- **Permissions:** Redirect already-authenticated users away from `/login`

#### D. Navigation & Routing
- **Route:** `/login`
- **Guard:** Redirect to role dashboard if already authenticated
- **Redirect on success:** Role-based (see above)

#### E. Implementation Notes
- **Folder:** `src/pages/auth/LoginPage.jsx`
- **Libraries:** `react-hook-form` for form state; `yup` for schema validation
- **Accessibility:** Fieldset + legend for radio group; `aria-invalid` on error fields; `autocomplete="email"` on email field
- **Security:** Never log credentials; clear form on unmount

#### B. UI Components
| Component | States |
|-----------|--------|
| `ShiftCard` | Default, pending-confirm (amber border), active (green), completed (gray) |
| `StatusBadge` | scheduled, confirmed, in-progress, completed, cancelled |
| `DateRangePicker` | Default, selecting |
| `ConfirmDialog` (cancel) | Closed, open, reason textarea |
| `GeolocationButton` (check-in/out) | Idle, requesting location, success, error |

#### C. Functional Elements
- **Filters:** Status select + date range; `GET /api/shifts/applicant/my-shifts?status=&page=`
- **Shift Detail:** `GET /api/attendance/shift/:shiftId` — fetch check-in/check-out record for a shift
- **Confirm Shift:** `PUT /api/shifts/:id/confirm` — only shown if status = `scheduled`
- **Cancel Shift:** Reason dialog → `PUT /api/shifts/:id/cancel` (shared endpoint; both employer & applicant role accepted)
- **Check-In:** `navigator.geolocation.getCurrentPosition()` → `POST /api/attendance/:shiftId/checkin { latitude, longitude }`
- **Check-Out:** Same with checkout endpoint
- **Loading:** Skeleton cards; geolocation spinner

#### D. Navigation & Routing
- **Route:** `/applicant/shifts`
- **Guard:** `ProtectedRoute role="applicant"`

#### E. Implementation Notes
- **Folder:** `src/pages/applicant/MyShiftsPage.jsx`
- **Libraries:** Browser Geolocation API (no third-party needed)
- **Accessibility:** Confirm dialogs trap focus; destructive actions (cancel) use red button

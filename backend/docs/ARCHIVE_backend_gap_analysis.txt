## 🔴 CRITICAL — Endpoint Does Not Exist in Backend

---

### FIND-01: `GET /api/applications/employer/recent` — Does NOT exist

| Property | Detail |
|----------|--------|
| **Used in docs** | SCR-17 Employer Dashboard — `GET /api/applications/employer/recent?limit=5` |
| **Backend reality** | No such route. The only employer application route is `GET /api/applications/job/:jobId` (applications for ONE specific job) |
| **Fix required** | On SCR-17 (Employer Dashboard) you cannot fetch recent applications across all jobs from a single call. **Options:** (A) Call `GET /api/jobs/employer/my-jobs` to get job IDs, then make parallel calls per job (expensive). (B) Use `GET /api/admin/dashboard/stats` recentApplications — but that's admin-only. **(Recommended):** The frontend should derive "recent applications" by fetching jobs first, then page-1 applications per job, or request the backend team add a `GET /api/applications/employer/recent` endpoint. |

---

### FIND-02: `GET /api/admin/jobs` — Does NOT exist

| Property | Detail |
|----------|--------|
| **Used in docs** | SCR-31 Admin Dashboard — `GET /api/admin/jobs?limit=5`; SCR-34 Moderate Jobs — `GET /api/admin/jobs?page=&status=&search=` |
| **Backend reality** | `adminRoutes.js` only has `DELETE /api/admin/jobs/:id`. There is **no GET** for admin jobs list. |
| **Fix required** | SCR-34 (Moderate Jobs) cannot list jobs. The public `GET /api/jobs` works without auth but lacks admin-level delete ability. **Recommended fix:** Use `GET /api/jobs` (public) for listing on SCR-34, and use the existing `DELETE /api/admin/jobs/:id` for deletion. |

---

### FIND-03: `PUT /api/admin/jobs/:id/close` — Does NOT exist

| Property | Detail |
|----------|--------|
| **Used in docs** | SCR-34 Moderate Jobs — `PUT /api/admin/jobs/:id/close` |
| **Backend reality** | Admin has NO close-job endpoint. Only `DELETE /api/admin/jobs/:id` exists. |
| **Fix required** | Remove "Close Job" button from SCR-34 admin view. Admin can only **delete** jobs, not close them. Closing is employer-only (`PUT /api/jobs/:id/close`). |

---

### FIND-04: `PUT /api/admin/applicants/:id/activate` — Does NOT exist

| Property | Detail |
|----------|--------|
| **Used in docs** | SCR-33 Manage Applicants — shows "Activate" toggle button calling `PUT /api/admin/applicants/:id/activate` |
| **Backend reality** | `adminRoutes.js` only has `PUT /api/admin/applicants/:id/deactivate`. There is **no activate** endpoint. |
| **Fix required** | Remove the "Activate" button from SCR-33. Admin can only **deactivate** applicants. The UI should show deactivated accounts as read-only with no reactivation option (or request backend to add the endpoint). |

## 🟠 HIGH — Wrong Path or Wrong Field Name

---

### FIND-05: Admin stats route is `/api/admin/dashboard/stats` not `/api/admin/stats`

| Property | Detail |
|----------|--------|
| **Used in docs** | SCR-31 and SCR-35 — both call `GET /api/admin/stats` |
| **Backend reality** | `adminRoutes.js` registers `router.get('/dashboard/stats', ...)` → actual path is `GET /api/admin/dashboard/stats` |
| **Fix required** | Update all API calls in SCR-31 and SCR-35 from `GET /api/admin/stats` to `GET /api/admin/dashboard/stats` |

---

### FIND-06: Reject application body field is `rejectionReason` not `rejectReason`

| Property | Detail |
|----------|--------|
| **Used in docs** | SCR-21 Reject Modal — `PUT /api/applications/:id/reject { rejectReason }` |
| **Backend reality** | `applicationController.js` line 298: `const { rejectionReason } = req.body;` — field is `rejectionReason` |
| **Fix required** | Change the reject modal's POST body field from `rejectReason` to `rejectionReason` in SCR-21 implementation |

---

### FIND-07: Shift cancel is available to BOTH employer and applicant

| Property | Detail |
|----------|--------|
| **Used in docs** | SCR-10 (applicant) documents `PUT /api/shifts/:id/cancel`; SCR-26 (employer) also documents it — but they are treated as separate concerns in the docs with different auth |
| **Backend reality** | `shiftRoutes.js` line 79–84: `PUT /:id/cancel` uses `authorize('employer', 'applicant')` — **same endpoint, both roles** |
| **Impact** | Minor — the docs are functionally correct in pointing to the same endpoint, but the frontend should be aware that both roles hit the same endpoint and the controller must validate ownership internally |

---

### FIND-08: Shift update endpoint (`PUT /api/shifts/:id`) is undocumented on frontend

| Property | Detail |
|----------|--------|
| **Backend reality** | `shiftRoutes.js` line 39–45: `PUT /:id` exists for employer to update shift details |
| **Used in docs** | SCR-26 (My Shifts Employer) says "Edit Shift → Opens edit modal" but does not specify which API endpoint it calls |
| **Fix required** | Add `PUT /api/shifts/:id` as the API call in SCR-26 Edit Shift modal functional elements |
